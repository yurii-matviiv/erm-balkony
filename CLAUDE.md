# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> Це наша спільна база знань по проєкту. Тримай її в актуальному стані: коли з'являється нова сутність, синхронізація чи архітектурне рішення — допиши сюди коротко, що і чому. Не переписуй сюди код — лише те, що не видно з самого коду (рішення, причини, статус, плани).

## Що це за проєкт

ERM — нова система для компанії balkony-vikna.kyiv.ua (вікна/балкони), яка замінює стару самописну PHP CRM ("balkonio_erm"). Стара система працює і досі використовується в продакшені — заміна відбувається поетапно:

1. Новий застосунок будується і тестується локально, працюючи зі своєю **власною** базою даних.
2. Стара база даних — **тільки джерело для одноразового перенесення даних** (users, clients, suppliers, candidates, ...), доступна через окреме read-only з'єднання `legacy`. Новий код НІКОЛИ не пише в це з'єднання.
3. На етапі деплою: піднімається чиста production-база нового застосунку, у `.env` змінюються `LEGACY_DB_*` на реальні продакшен-креди старої CRM (це лише env, без змін коду), і через адмінку "Синхронізація" заново проганяються всі мапери — це наповнює реальну production-базу.
4. Планується ~тиждень паралельної роботи обох систем, після чого стара БД повністю виводиться з експлуатації.

## Команди

```bash
php artisan serve                                   # локальний сервер (або Laragon vhost erm.balkony-vikna.kyiv.ua.test)
php artisan migrate                                 # застосувати міграції
php artisan shield:generate --all --panel=admin      # перегенерувати права/політики Filament Shield — ОБОВ'ЯЗКОВО після додавання нового Resource/Page
php artisan shield:super-admin                       # призначити роль super_admin користувачу
vendor/bin/pint                                      # форматування коду (Laravel Pint)
composer test                                        # php artisan config:clear + php artisan test
php artisan sync:legacy                             # ручний запуск повної синхронізації зі старої БД (всі мапери)
php artisan schedule:work                           # локальне тестування планувальника (тримає процес, кожну хвилину)
```

Команди виконує користувач у своєму терміналі (Windows PowerShell в Cowork-сесії дозволений лише на клік, без вводу тексту) — Claude дає точну команду, користувач її запускає і присилає вивід.

## ⚠️ При переїзді на хостинг: налаштування crontab

Після деплою на сервер **обов'язково** додати до crontab (один запис, одна хвилина):

```
* * * * * cd /path/to/erm.balkony-vikna.kyiv.ua && php artisan schedule:run >> /dev/null 2>&1
```

Це активує Laravel-планувальник. Зокрема він запускатиме `sync:legacy --scheduled` кожну хвилину — але тільки якщо авто-синхронізацію увімкнено через тумблер на сторінці "Синхронізація зі старої БД". Після остаточного переходу на нову систему тумблер вимикається, і синхронізація автоматично зупиняється (без змін у crontab).

**Інтервал авто-синку — теж налаштування** (`sync_interval_minutes` в app_settings, кнопка "Інтервал: N хв" на сторінці синхронізації): cron тікає щохвилини, але `--scheduled`-запуск тихо виходить, поки з часу останнього ЗАВЕРШЕНОГО проходу минуло менше N хвилин. Кожен прохід — повне перечитування всіх старих таблиць (інкрементального режиму свідомо немає — у старій БД немає надійного updated_at, а по id пропускались би зміни статусів). 1 хв — режим живої роботи менеджерів, 60 — тихий режим розробки. Ручний запуск (кнопка/`sync:legacy` без прапорця) інтервал ігнорує.

Шлях `/path/to/erm.balkony-vikna.kyiv.ua` замінити на реальний шлях на сервері (наприклад `/var/www/erm.balkony-vikna.kyiv.ua`).

## Архітектура синхронізації зі старою БД

Це центральна, найскладніша частина проєкту. Файли: `app/Services/Sync/`.

- **`Contracts/SyncMapper`** — інтерфейс одного "стара таблиця → нова таблиця" мапера.
- **`AbstractSyncMapper`** — спільна логіка (пагінація, підрахунок, ідемпотентний upsert по `legacy_id`, чанкований `run()`). Конкретний мапер реалізує тільки `transformRow()` (і опційно `afterUpsert()`).
- **`SyncMapperRegistry::all()`** — список усіх маперів; сторінки `SyncOverview`/`SyncTableDetail` будують UI з нього автоматично. Щоб додати нову пару таблиць — пишеш клас у `Mappers/` і додаєш один рядок у реєстр.
- Кожна нова таблиця, яку синхронізуємо, повинна мати nullable unique колонку `legacy_id` — без неї `AbstractSyncMapper` впаде (так і сталося з `candidates`, виправлено окремою міграцією `add_legacy_id_to_candidates_table`).
- **Два старих джерела в одну нову таблицю** (`CandidatesSyncMapper` з `candidates` + `AddCandidateSyncMapper` з `add_candidate`, обидва пишуть у нову `candidates`): щоб уникнути конфлікту id (обидві старі таблиці мають свою нумерацію з 1), `AddCandidateSyncMapper` зсуває свій `legacy_id` на `+1_000_000` і перевизначає `resolveLegacyId()`/`syncedQuery()` з `AbstractSyncMapper` (ці два hook-методи додані спеціально для такого випадку).
- Один старий рядок може породжувати **кілька** нових записів у різних таблицях через `afterUpsert($oldRow, $newId)` — приклади: `UsersSyncMapper` призначає роль; `ClientsSyncMapper`/`SuppliersSyncMapper` нічого зайвого не створюють; `CandidatesSyncMapper`/`AddCandidateSyncMapper` додатково створюють/оновлюють пов'язаний `vacancy_applications` запис.
- Поточні мапери: `UsersSyncMapper`, `ClientsSyncMapper`, `SuppliersSyncMapper`, `CandidatesSyncMapper`, `AddCandidateSyncMapper`.

### Одноразові, нерегулярні скрипти переносу (НЕ мапери)

`app/Console/Commands/MoveSupplierContactsFromClients.php` — переніс 47 старих "клієнтів" з `caller_type='supplier'` у `supplier_contacts`, прив'язавши кожен запис до конкретного постачальника **вручну прочитаним** мапінгом (назва компанії була вписана прямо в текстове поле "ім'я", напр. "Водій Алнат"). Це НЕ повторюваний мапер — мапінг хардкоджений по конкретних `legacy_id` клієнтів. **Перед фінальним деплоєм** потрібно перевірити, чи з'явилися в продакшн-БД нові клієнти з `caller_type='supplier'` після дати цього аналізу — їх цей скрипт не підхопить автоматично.

## Чому деякі сутності розділені (а не "типи" одної)

Свідоме архітектурне рішення: **Client, Supplier, Candidate — три різні сутності**, не варіанти одної "контактної" таблиці. Причина: у старій системі `clients.caller_type` змішував реальних клієнтів, постачальників, спам і випадкові дзвінки в одній таблиці — це визнана помилка старої моделі, яку свідомо не повторюємо.

- **Supplier** має `hasMany` дітей: `SupplierContact` (контактні особи) і `SupplierPaymentProfile` (платіжні реквізити — у постачальника може бути кілька платників/ФОПів). Обидва редагуються інлайн через `Repeater` у формі постачальника, **не** окремі ресурси/relation managers.
- **Vacancy / Candidate / VacancyApplication** — теж три окремі сутності: Vacancy = довідник посад (3 рядки: Монтажник/Помічник/Менеджер — все, що було в старій системі), Candidate = людина-кандидат (телефон навмисно НЕ unique — дублікати вирішує людина через пошук+вибір при створенні заявки), VacancyApplication = одна конкретна заявка (хто, на яку вакансію, звідки, чи цільова).
- Аналогічна, ще не побудована сутність: **наші власні юридичні особи/ФОПи**, на які виставляються рахунки клієнтам (структурно схожа на `SupplierPaymentProfile`, але прив'язана до менеджера/компанії, а не до постачальника) — узгоджено концептуально, код не писався.

## `clients.name` видалено — старе ім'я тепер у `first_name`

Раніше `ClientsSyncMapper` копіював старе єдине поле `name` у НОВУ окрему колонку `clients.name`, а `Client::getFullNameAttribute()` мовчки підставляв її, якщо структуровані `last_name`/`first_name`/`middle_name` порожні. Користувача це сильно збентежило: інтерфейс показував правильне ім'я, а `php artisan tinker` на `->only(['last_name','first_name'])` показував `null` — складно зрозуміти, звідки насправді береться відображене значення.

**Рішення:** колонки `clients.name` більше не існує. Migration `move_name_to_first_name_on_clients_table` (1) переносить старі значення `name` → `first_name` (тільки де `first_name` ще порожнє — щоб не затерти вже введені нові дані), (2) дропає колонку. `ClientsSyncMapper` тепер пише старе `name` напряму в `first_name`. **Розбиття на прізвище/ім'я/по-батькові — НЕ зроблено навмисно** (окрема майбутня задача за бажанням користувача) — для старих клієнтів усе старе ім'я просто лежить одним рядком у `first_name`, без жодного "прихованого" поля-дублікату. `Client::getFullNameAttribute()` спрощено — більше не має фолбеку на `name`, просто склеює структуровані поля (або "Без імені").

Зачищено всі інші місця, що посилались на стару колонку: `ClientResource` (форма + `searchable()`), `LeadResource` (двічі `searchable()`), `MoveSupplierContactsFromClients` (`$client->name` → `$client->full_name`, бо команда теоретично може запуститись ще раз перед фінальним деплоєм).

## Filament: два паттерни ресурсів

1. **Повний ресурс зі сторінками** (`UserResource`, `ClientResource`) — стандартні `index`/`create`/`edit` routes.
2. **Модальний ресурс** (`SupplierResource`, `VacancyResource`, `VacancyApplicationResource`) — у `getPages()` зареєстрована **тільки** `index`. Filament тоді сам відкриває `CreateAction`/`EditAction` як модалку замість переходу на сторінку. Обраний свідомо для простих сутностей, де повноцінна сторінка — оверкіл (за прямим запитом користувача). Дочірні колекції (контакти/платіжні профілі постачальника) редагуються через `Repeater::make(...)->relationship()` прямо в тій же модалці.

Для "заявка на вакансію" обрано ще один паттерн: `Select::make('candidate_id')->relationship(...)->searchable([...])->createOptionForm([...])` — пошук існуючого кандидата за телефоном/ПІБ або створення нового прямо в модалці. Це і є той "чи кандидат вже є в базі" чек, без жодної кастомної JS/reactive-логіки.

## Паролі користувачів — переносяться зі старої CRM

Стара CRM хешує паролі через `password_hash(..., PASSWORD_DEFAULT)` (перевірено в `dev.ERM-btv/app/Controllers/AccountController.php`) — це той самий bcrypt, що й у Laravel, тож `UsersSyncMapper` копіює хеш як є, і люди входять у нову систему зі своїми звичними паролями. Два важливі правила:

1. Хеш копіюється, лише якщо починається з `$2` (bcrypt); інакше — випадковий непридатний пароль.
2. `password` виключено зі списку колонок, які апдейтяться при повторному синку (`UsersSyncMapper::persistRow()`) — інакше щохвилинний авто-синк перезаписував би паролі, змінені вже в новій системі (ця помилка реально вистрелила при першому деплої: мапер тоді генерував випадковий пароль на КОЖНОМУ запуску).

Масова розсилка "встановіть пароль" не потрібна. Точкова зміна пароля — поле "Пароль" у формі користувача.

## Ролі та права

`spatie/laravel-permission` + `bezhansalleh/filament-shield`. `super_admin` — повний доступ (gate-before). Решта ролей береться зі старих `atom_roles` (12 шт.) автоматично під час синку користувачів, але створюються **з нулем прав** — це свідома страховка, права роздаються вручну пізніше через сторінку Roles. Після додавання будь-якого нового Resource/Page — обов'язково `shield:generate --all --panel=admin`, інакше Filament Shield не знає про нову сутність.

## Модуль "Заявки" (Lead) — найскладніший, у розробці поетапно

Стара база вже мала готову таблицю `leads` (14 117 рядків!) — звідти взято реальні довідники замість вигадування з нуля:

- `leads.order_type` (реальний розподіл) → нова таблиця `lead_service_types`, посівні значення: Вікна, Балкони, Вікна в котеджі, Ремонт вікон, Заміна склопакета, Обшивка балкона, Балкон з виносом, Сітки, Балкон під ключ, Підвіконня, Внутрішні ролети. Many-to-many з Lead (`lead_lead_service_type`) — один лід може мати кілька послуг одночасно.
- `leads.status` у старій системі звалював робочі стадії, причини відмови і технічні позначки в одну купу (new/processing/zamir/accepted/canceled/not_targeted/another_city/...). У новій системі це свідомо розділено на три поля: `stage` (воронка, завжди вперед: new→contacted→measurement_scheduled→measurement_done→proposal_sent→negotiation→closing), `status` (open/won/lost) і `lost_reason` (вільний текст, тільки коли lost). Список опцій — `Lead::stageOptions()`/`statusOptions()`.
- `leads.source` (call/office-visit/...) → нове поле `source` на Lead, але поки тільки 3 варіанти ручного введення: Дзвінок, Візит в офіс, Передали контакт. Інші старі значення (utm_*, site_source, gdid тощо) — це маркетингова атрибуція сайту/реклами, свідомо НЕ перенесено — буде окремий майбутній модуль "сайт" (автоматичні заявки з UTM/Google Ads).
- Старе `another_appeal` (одне зі значень order_type) — це не послуга, а позначка "повторне звернення". У новій системі цю роль виконує `Lead.application_type` (new/repeat), і обчислюється **автоматично**: "repeat", якщо в обраного клієнта вже є хоч одна попередня заявка В НОВІЙ системі (не "клієнт існує в БД" — 12k+ вже синхронізованих старих клієнтів зробили б це марним).

**Lead не дублює контактні дані** — `belongsTo Client` (як Candidate). `Client` доповнено структурованими полями (`last_name`/`first_name`/`middle_name`, `street`/`house_number`/`apartment_number`/`floor`/`city` — місто за замовч. "Київ"), поряд зі старими `name`/`address` (для вже синхронізованих клієнтів). `Client::getFullNameAttribute()`/`getFullAddressAttribute()` показують структуровані поля, якщо є, інакше — старі.

Пошук клієнта в формі ліда — `Select::make('client_id')->relationship(...)->searchable(['phone',...])->createOptionForm([...])`, той самий патерн, що й Candidate в заявках на вакансії (свідомо обраний користувачем як "перевірений", а не кастомний live-typeahead).

Захист від дублів: один Client — один Lead на календарний день (валідація `rule()` на полі client_id). Якщо клієнту потрібно кілька різних послуг одного дня — додаються через мульти-select `serviceTypes` в той самий лід, а не другим лідом.

**Синхронізація історичних лідів (LeadsSyncMapper) — реалізовано, повна, не часткова.** Рішення (підтверджено користувачем): синхронізуємо одразу і сам лід, і маркетингові дані — дані вже є в старій БД, не використати їх зараз дешевше, ніж колись довантажувати окремо.

Архітектурне рішення, яке з цього випливає — **розділення на 3 таблиці** замість однієї "плоскої", як було в старій системі:

- **`leads`** (сам лід) — те, що бачить менеджер у формі.
- **`lead_marketing_data`** (1:1, через `lead_id`) — utm_*, device, ip_address, user_fingerprint, client_cookie_id, gclid (старе поле звалося `gdid`)/gbraid/wbraid, site_source (btv/avicon — на який з сайтів/брендів прийшла заявка), form_name/form_position, referrer, referral_1-4. Заповнюється лише автоматично (синком зараз, і майбутнім модулем "сайт" — нічого з цього менеджер вручну не торкається).
- **`lead_calls`** (1:багато, на майбутнє) — лог дзвінків Binotel. Схема створена, **даних і UI немає** — в старій БД підтверджено відсутня окрема таблиця дзвінків (`get_call_binotel`/`binotel_chat` — це лише значення `leads.source`, без тривалості/запису/ID дзвінка). Чому окремою таблицею, а не в marketing_data: один лід може мати кілька дзвінків з часом (це лог подій), а marketing_data — одноразовий знімок при створенні.

**Що НЕ перенесено зі старих `leads` свідомо:**
- `profile` (rehau_60/70/80 — профіль вікна) і `advance_calculation1-3` (попередній розрахунок ціни) — це вже атрибути замовлення/комерційної пропозиції, не ліда. Дані лишаються доступні в старій БД на час паралельної роботи; перенесемо, коли дійдемо до модуля "Замовлення".
- Рядки зі старим `caller_type` в ('spam','other','supplier') — пропускаються повністю, це не справжні ліди (той самий принцип, що і для Client/Supplier розділення).
- Старе поле `date` (окреме від created_at) — призначення не до кінця зрозуміле з самих даних, не мапилось нікуди, щоб не вигадувати неправильну семантику.

**Мапінг старого `status` (12 значень, де все було в одну купу) → нові `stage`/`status`/`lost_reason`** — лосі (lossy) симпліфікація навмисно: повне старе значення зберігається у `leads.legacy_status` без втрат, на випадок якщо мапінг знадобиться переглянути. Деталі мапінгу — `LeadsSyncMapper::STATUS_MAP`.

`application_type` (нова/повторна) для історичних лідів обчислюється так само, як і в живій формі: чи в клієнта вже була РАНІШЕ заявка. Синхронізація йде по зростанню старого id (хронологічно), тому це коректно відтворює історію.

**Важливо: порядок синхронізації має значення.** LeadsSyncMapper резолвить `client_id`/`manager_id` через `legacy_id` — спочатку мають бути синхронізовані Users і Clients, інакше ліди без знайденого клієнта просто пропускаються (лічильник skipped).

**Що ще НЕ зроблено в цьому модулі** (наступні етапи): сторінка "Воронка лідів" (візуалізація, аналогічно "Воронці вакансій"), реальна інтеграція Binotel (зараз лише порожня схема `lead_calls`), поля джерела з сайту/реклами в самій формі створення (зараз тільки 3 ручні джерела; `lead_marketing_data` вже готова приймати дані, коли модуль сайту з'явиться), модуль "ресурс" (визначення відповідального), власна маска вводу телефону (зараз — стандартний Filament Select, без кастомної маски).

## Платежі — принципи (ЦЕЙ розділ — точка істини; зміна логіки платежів починається зі зміни ЦЬОГО розділу)

Домовленість процесу (за явним запитом користувача): будь-яка зміна логіки платежів спочатку фіксується тут, потім реалізується в коді. Не дублювати ці принципи в інших розділах/файлах — посилатись сюди.

**Принцип 1 — жодних зашитих цифр.** Всі фінансові показники рахуються ВИКЛЮЧНО з записів платежів (`order_payments` + `expenses`). Стара CRM мала в коді константи (+9000 оренда, +12600, +6294, +30000, +2% від доходу прямо в `CashoutsController`) — це ЗАСТАРІЛИЙ шар: реальні платежі на ці суми вже існують у базі як записи (перевірено: 9000 = оренда офісу, вже видно в "Аналітиці рахунків"). Константи НЕ переносимо і не відтворюємо.

**Принцип 2 — перенесення ≠ копіювання.** Задача міграції старих платежів — не перелити таблицю 1:1, а РОЗКЛАСТИ платежі в нову чисту структуру (категорія/підкатегорія/рахунок/платник). Класифікація: (а) чіткі правила мапінгу по полях; (б) коментар платежу — ОСТАННІЙ аргумент і ТІЛЬКИ на етапі міграції; (в) що не розклалось автоматично — позначається як нерозібране і розбирається вручну через інтерфейс.

**Принцип 3 — нові платежі народжуються правильними.** У новій системі платіж створюється одразу повністю структурованим. Ніякої логіки "вгадай по коментарю" для нових записів — коментар лише довідковий текст.

**Принцип 4 — авторство.** Кожен платіж має `created_by`. У старій базі це `orders_payments.user_id` (коментар колонки: "хто редагує") — мапиться через `users.legacy_id`, тож авторство зберігається і для історичних платежів.

**Реалізовано (перша ітерація):**

- `created_by` + `classification_status` в обох таблицях (міграція `add_created_by_and_classification_to_payments`). Мапери заповнюють: авторство через старий `user_id`→`users.legacy_id`; класифікація за правилами — `OrderPaymentsSyncMapper::classify()`: client-рядки чисті самі по собі (контрагент визначається через замовлення — в старій БД `user_id_by_type=0`); для installer/gauger/client `payer_name` резолвиться фолбеком через замовлення (installer_id/surveyor_id/client_id), supplier — тільки пряме ім'я; `expense`+`salary` — classified (зарплатні рядки для майбутнього модуля "Зарплата"); `between_accounts` завжди classified; + відомий метод. СУМА — НЕ критерій (явний фідбек користувача: unsorted означає "система не розуміє, куди це", а скасований рахунок з нулем — цілком зрозумілий запис). `GeneralExpensesSyncMapper::classify()` — потрібні І категорія, І підкатегорія. Решта — `unsorted` (реальна ручна черга: постачальники без назви, невідомий метод тощо).
- **SQL view `payments_ledger`** (`order_payments` UNION ALL `expenses`, нормалізовані колонки, синтетичний ключ `op-N`/`ex-N`) + read-only модель `PaymentLedgerEntry`. При додаванні колонок у будь-яку з таблиць view треба перестворити новою міграцією.
- **Сторінка "Платежі"** (`app/Filament/Pages/Finance/Payments.php`, група "Фінанси"): комбінований фільтр (період з пресетами включно з "попередній місяць", напрям, джерело, категорія+залежна підкатегорія, метод, статус, розбір, тип платника, постачальник (по payer_name), автор, №замовлення) + пошук по коментарю/платнику. Рядки замовлень — read-only з посиланням на замовлення (редагуються в майбутньому модулі "Оплати"); загальні витрати — створення/редагування/видалення прямо тут (редагування з повними категорією+підкатегорією автоматично переводить рядок у classified — це і Є ручний розбір черги "Не розібрані").
- `ExpenseResource` ("Витрати") видалено — його форма створення переїхала в дію "Додати витрату" на "Платежах".

**Група "Анульовано" (void, not delete)** — рішення користувача: платежі, які системі точно не потрібні (напр., дублі від банківського бота, який писав серію рахунку "9126" в order_id замість номера замовлення, або платежі до несинканих замовлень), НЕ пропускаються синком і НЕ видаляються. Вони імпортуються з `order_id = NULL` (колонка стала nullable — міграція `make_order_id_nullable_on_order_payments`), `status = 'canceled'` (аналітика рахує тільки received → у підсумки не входять) і `classification_status = 'annulled'` — окрема група у фільтрі "Розбір" на "Платежах": історію видно, за потреби можна розібрати вручну. Перепідв'язку таких платежів до реальних замовлень (номер часто видно в коментарі: "рахунок №9126-9126-2196" → замовлення 2196) свідомо ВІДКЛАДЕНО, поки не спроєктований workflow перепривʼязки. `classification_status` виключено зі списку колонок повторного синку в ОБОХ маперах платежів (як password у користувачів) — ручний вердикт людини переживає щохвилинний авто-синк.

**Рахунок (ФОП) на платежах — реалізовано.** Довідник = існуючі `privatbank_accounts` ("Акаунти ПриватБанк", три ФОПи: Рабченюк/Смірнова/Пижанова, привʼязані до users). `order_payments.fop_account_id` + заповнений `expenses.fop_account_id`; старий `fop_account` у старій БД — це old users.id власника ФОПа (35/47/60; NULL/0 — не вказано; 1, 68 — менеджери без власного ФОПа). Правило резолву (рішення користувача): явний маркер → ФОП того юзера; інакше → ФОП МЕНЕДЖЕРА замовлення; немає і в нього → NULL "не визначено". Готівка/безготівка — окрема вісь (payment_method), ФОП відповідає "чий облік". На "Платежах": колонка + фільтр "Рахунок (ФОП)", вибір ФОПа у формі витрати. `payments_ledger` перестворено (v2, migration recreate_payments_ledger_view_with_fop).

**Сторінка "Платіжні правила"** (`app/Filament/Pages/Finance/PaymentRules.php`, Фінанси) — контрольна панель "як платежі заходять і розкладаються", за явним запитом користувача (адмін бачить механіку без розробника; розробнику — точка істини в системі). Блок 1: міграція зі старої CRM — read-only fieldMap() трьох платіжних маперів з лічильниками + підсумки розбору (розібрані/не розібрані/анульовані). Блок 2: таблиця `payment_rules` — редаговані правила розподілу (поле/умова/шаблон → категорія/підкатегорія, пріоритет, тумблер) для МАЙБУТНЬОГО імпорту банківських транзакцій; поки їх ніщо не виконує, посіяне правило Google Ads — еталон форми. Зверху — PageDoc `payment-rules/principles` (посіяний міграцією, дзеркало розділу "Платежі — принципи").

**Мапер `google_ads_pay`** — стара таблиця автосписань Google Ads з карток (по суті міні-журнал банківських транзакцій: num_doc = унікальний банківський документ, pay_account = IBAN). Іде в `expenses` як marketing/`google_ads` ("Google Ads (списання)") — ОКРЕМА підкатегорія від ручних платежів підряднику (`google`), за явним запитом. legacy_id зсунуто на +2 000 000 (той самий прийом, що AddCandidateSyncMapper). ФОП — прямий матч IBAN → privatbank_accounts.account_number. Звірка "ПриватБанк — Google Ads" в аналітиці тепер порівнює ПБ саме з підкатегорією google_ads.

Ще не зроблено: імпорт банківських транзакцій, який виконуватиме payment_rules (по-транзакційна звірка — схема обговорена 2026-07-07); класифікація order-рядків через UI; workflow перепривʼязки анульованих платежів до замовлень.

## Статус-бар і "заявка на замір" на сторінці ліда

Перший крок руху ліда до майбутнього модуля "Замовлення". `EditLead::getHeader()` (рендериться автоматично над формою, той самий механізм, що й `getFooter()` для документації) додає:

1. **Клікабельний статус-бар** — рядок кнопок, одна на етап з `Lead::stageOptions()`, поточний підсвічений кольором (`bg-primary-600`). Клік одразу викликає `setStage()`, який оновлює `stage` і робить `$this->fillForm()`, щоб звичайний Select "Етап" у формі нижче не лишився зі застарілим значенням.
2. **Action-бар** — рівно ОДНА кнопка "що робити далі", не статус. Зараз реалізований лише перший крок — "Створити заявку на замір" (`EditLead::createMeasurementAction()`). Наступні кроки (КП, узгодження тощо) свідомо не реалізовані — додаються по одному, як і попросив користувач.

**`lead_measurements`** (lead_id, scheduled_date, scheduled_time, surveyor_id, installer_id, comment) — нова дочірня таблиця. Заявка на замір призначає бригаду: замірника і монтажника (можуть бути однією людиною). **Важливе бізнес-правило, яке легко забути:** система завжди орієнтується на ЗАМІРНИКА як на відповідального за бригаду, навіть якщо монтажник інший — це явно зафіксовано в коментарях миграції і моделі (`LeadMeasurement::surveyor()`), знадобиться в модулі "Замовлення"/зарплата пізніше. Поки немає формальної ролі "Замірник" — вибір просто з усіх `User`, без фільтра по ролі (додати фільтр пізніше, коли роль з'явиться).

Lead може мати кілька заявок на замір з часом (перенесення замірів — нова заявка, не редагування старої) — `Lead::latestMeasurement()` показує останню.

Перейменував останній етап воронки: `closing` тепер має лейбл "Конвертовано в замовлення" (було "Оформлення угоди") — той самий ключ у БД, просто точніша назва під майбутній модуль Замовлення.

## Клієнт vs. Лід — дві різні адреси, розділена форма редагування

Користувач сам сформулював ключове розрізнення під час перегляду сторінки ліда: **Client — це людина, яка існує незалежно від конкретного звернення** (може мати 0 лідів), а **адреса об'єкта в конкретному ліді може відрізнятись від домашньої/контактної адреси клієнта** (інша нерухомість, родич, офіс тощо). Тому:

- `leads` отримав власні `site_city`/`site_street`/`site_house_number`/`site_apartment_number`/`site_floor` (міграція `add_site_address_to_leads_table`) — окремо від адресних колонок `clients`. Префікс `site_` навмисно (не `address_`), щоб не плутати з полями клієнта поруч у коді. `Lead::getSiteAddressAttribute()` — те саме, що `Client::getFullAddressAttribute()`, але для адреси об'єкта; повертає `null`, якщо нічого не заповнено (НЕ підставляє адресу клієнта замість — це б замаскувало відсутність даних).
- `EditLead::form()` тепер **окремий** від `LeadResource::form()` (раніше форма ліда була одна на створення+редагування). Причина: на створенні (модалка) вибір/створення клієнта через `client_id` Select з пошуком — це нормально. На редагуванні ВЖЕ існуючого ліда той самий Select виглядав як "додати клієнта" і дозволяв випадково перепідʼєднати лід до іншої людини через пошук — заплутувало користувача.
- На сторінці редагування ліда клієнт показується **тільки на читання** (`TextEntry`: ПІБ, телефон, адреса клієнта) у секції "Клієнт" з кнопкою-дією "Редагувати клієнта" в заголовку секції (`Section::headerActions()`) — відкриває модалку з повними полями `Client` і зберігає напряму в `clients`, а не в `leads`. Це свідомо: клієнт може мати кілька лідів, тож редагування тут міняє людину всюди, а не тільки в межах цього звернення.
- Секція "Адреса об'єкта" — окремі інпути `site_*`, з підписом, що це може відрізнятись від адреси клієнта вище.
- На десктопі жодне ОДИНОЧНЕ поле (не картка-секція) не повинно займати 100% ширини — максимум 50%, на мобільному — 100%. Якщо секція має `->columns(2)` і поле без `->columnSpanFull()`, це виходить само собою. Для секції "Адреса об'єкта" (де полів навмисно по одному в рядку, без `->columns()`) довелось обмежити ширину інакше — через `->extraFieldWrapperAttributes(['class' => 'lg:max-w-[50%]'])` на кожному полі.
- **Картки (Section) на сторінці завжди мають бути одна під одною, на 100% ширини рядка** — навіть якщо `Filament\Resources\Pages\EditRecord::defaultForm()` примусово ставить `$schema->columns(2)` на будь-якій схемі, яка сама не визначила свій `columns()` (vendor-код, не наша помилка). Тому в `form()` будь-якої кастомної edit-сторінки з кількома Section-картками ОБОВ'ЯЗКОВО викликати `$schema->columns(1)` на верхньому рівні — інакше картки намагаються стати по дві в ряд на широких екранах. Застосовано в `EditLead` і `EditOrder`.

## Замовлення (Order) — друга стадія після Ліда

Створюється виключно через дію "Створити замовлення" на сторінці ліда (`EditLead::createOrderAction()`) — у `OrderResource` НЕМАЄ create-сторінки/кнопки навмисно, щоб замовлення завжди починалось від реального клієнта+адреси, а не вводилось вручну з нуля. Дія:

1. Показується в action-барі ліда (`lead-status-bar.blade.php`) ОДРАЗУ ПІСЛЯ заміру — поки замовлення не існує. Як тільки замовлення створене, кнопка зникає, замість неї — посилання "Перейти до замовлення".
2. Перед створенням питає підтвердження (`->requiresConfirmation()`), у тексті підтвердження — реальні ПІБ клієнта й адреса об'єкта (а не загальне "Ви впевнені?"), щоб менеджер міг помітити помилку клікнув не туди.
3. Копіює `client_id`, `manager_id`, адресу (`Lead::site_address` або, якщо порожньо, адресу клієнта) і бригаду з останньої заявки на замір (`surveyor_id`/`installer_id` — той самий замірник-відповідальний, що і в `LeadMeasurement`).
4. Переводить лід у `stage = 'closing'` і переадресовує на сторінку нового замовлення.

**Структура `orders`** — поля й назви взяті безпосередньо зі старого шаблону відображення (`dev.ERM-btv/app/Views/Order/order.php` + `blocks-of-orders/*.php`, прочитано read-only, НІЧОГО там не редагувалось) і старої таблиці `orders` (~2.1к рядків). Свідомо НЕ перенесено в цей перший прохід (див. докблок міграції `create_orders_table`):
- по-платіжні рядки (стара `orders_payments`) — майбутній модуль "Оплати".
- завантажені файли (стара `orders_files`, інтеграція з Google Disk) — не плануємо повторювати Google Disk, стратегія файлів ще не вирішена.
- блоки комерційної пропозиції/рахунку постачальника в UI — майбутні "Рахунки"/"Оплати" (самі $-поля, що їх живили, залишені як звичайні дані).
- живий JS-калькулятор зарплати монтажника (відсоток від вартості) — бізнес-логіка майбутнього модуля "Зарплата", тут це просто звичайні числові поля.

`stage`/`status` — той самий підхід, що і в Lead: стара система мала ОДНЕ числове `status_new` (1-10, довідник `statuses`) ПЛЮС окремий старіший текстовий `status` enum, що з часом розійшлися. Обидва зберігаються вербатим (`legacy_status`/`legacy_status_new`) для трасування; новий `stage` — чиста переробка сенсу `status_new`, а `status` — лише open/done/cancelled. `OrdersSyncMapper` переводить `client_id` (обов'язково, без матчу — рядок пропускається), `lead_id`, `manager_id`, `consultant_id`, `installer_id`, старий `gauger_id` → `surveyor_id` (та ж роль "відповідальний за бригаду", що й у `LeadMeasurement`), `supplier_id` — усе через `legacy_id` відповідних таблиць.

`OrderResource`/`EditOrder` — повна сторінка (не модалка, як і Lead), картки: Клієнт (read-only), Замовлення, Реквізити, Дати, Вартість, Коментарі. Той самий принцип "жодне поле не на 100% ширини на десктопі" і `$schema->columns(1)` на верхньому рівні, що і в EditLead — див. вище.

**Хитрість з `->fillForm(Closure)` у Section header-action — не підтягувало дані.** У `HasEditClientAction::editClientAction()` модалка "Редагувати клієнта" відкривалась з правильними полями/лейблами, але ВСІ значення були порожні (крім статичних `->default()`, напр. "Київ"), хоча сама секція поруч коректно показувала ПІБ/телефон клієнта. Причина не до кінця діагностована через сам код (немає змоги виконати PHP в цій сесії), але достеменно НЕ через прив'язку `$this` — `->action(function (array $data) { $this->getRecord()... })` в тому самому файлі й сусідніх (`createMeasurementAction` тощо) працює надійно. Виправлення: замість `->fillForm(fn (): array => $this->getRecord()->client?->only([...]))` — резолвити масив ЗАЗДАЛЕГІДЬ (звичайний `array`, не `Closure`) прямо в тілі `editClientAction()`, до `Action::make(...)`. Метод однаково перевиконується щоразу, коли Filament перебудовує `form()` (зокрема прямо перед монтуванням дії), тож "свіжість" даних не страждає. **Якщо в майбутньому знову з'явиться "модалка з правильними полями, але порожніми значеннями" — спочатку пробувати саме цей фікс** (eager-array замість Closure у `fillForm()`), а не шукати проблему в білдингу/ролях/кешуванні.

**`App\Filament\Concerns\HasEditClientAction`** — спільний трейт (`editClientAction(): Action`), винесений із колишнього дубльованого методу в `EditLead`, тепер використовується і в `EditLead`, і в `EditOrder` (Section "Клієнт" → `->headerActions([$this->editClientAction()])`, олівець біля заголовка картки). Припускає, що `$this->getRecord()` має звʼязок `client()` (правда для Lead і Order). **Свідомо НЕ обмежений роллю** — на відміну від `canReassignOwnership()` нижче: користувач явно попросив, щоб менеджер, навіть не маючи доступу до загального меню "Клієнти", міг виправити дані КОНКРЕТНОГО клієнта прямо з картки ліда/замовлення через це модальне вікно — редагування контактних даних клієнта це інша за вагою дія, ніж перепризначення відповідального.

**Поля "Менеджер" і "Консультант" приховані від ролі "Менеджер"** (`EditOrder::canReassignOwnership()`) — менеджер бачить, хто відповідальний (завжди видно в header-барі вище), але не може це змінити; дозволено лише "Керівник відділу продажу"/"Адмін"/`super_admin`. ВАЖЛИВО: перевірка йде через `auth()->user()->getActiveRoleName()` (роль, яку зараз ПЕРЕГЛЯДАЄ користувач через перемикач ролі в сайдбарі), а НЕ через `hasRole()` (реальні ролі в БД) — інакше адмін, що теж має роль "Менеджер", бачив би поле навіть під час перегляду інтерфейсу очима менеджера, що зводить нанівець сенс цього перемикача. Це перший у проєкті приклад role-gated поля у формі ресурсу — той самий паттерн (`->hidden(fn () => ! in_array(auth()->user()?->getActiveRoleName(), [...], true))`) можна повторювати для інших полів, де потрібне обмеження за роллю.

**Дружніший вигляд за зразком старого інтерфейсу** (користувач показав скрін старої CRM і попросив зробити схожіше, не 1:1): `EditOrder::getHeader()` додає компактний summary-бар (іконка + "Замовлення №X", бейдж етапу кольором за статусом, бейдж типу, рядок менеджер/дата створення/консультант/посилання на лід) — `order-header.blade.php`. У `form()` секції отримали `->icon()` + `->compact()`; секція "Замовлення" перебудована через вкладені `Filament\Schemas\Components\Grid` (по 2-3 поля в рядок: адреса+тип, менеджер+консультант, монтажник+замірник, м²+тоглі), секція "Дати" — теж по 3 поля в рядок, секція "Вартість" — `->columns(3)`. Це не суперечить правилу "жодне поле не 100% на десктопі" — навпаки, посилює його (поля тепер по 2-3 в рядок замість 1-2).

**Чому в формі НЕМАЄ полів "Постачальник", "Рахунок від постачальника", "Оплачений рахунок постачальнику"** (хоча колонки `supplier_id`/`invoice_from_supplier`/`paid_invoice_to_supplier` і звʼязок `Order::supplier()` існують) — свідоме рішення після пояснення користувача: один picker/одне номер-поле на все замовлення не відповідає реальності, бо в одному замовленні може бути кілька постачальників, і "хто постачальник"/"який рахунок" по суті визначає кожен окремий "Рахунок від постачальника" (інвойс), а не замовлення в цілому. Правильна модель — ці поля на майбутній сутності "інвойс/рахунок" (модуль Рахунки/Оплати), з можливістю підтягнути дані з самого інвойсу (за аналогією з тим, як користувач описав свій досвід з імпортом інвойсів у Норвегії — підтягується, що є, решта дозаповнюється вручну), а НЕ flat-поля на Order. Самі колонки/звʼязок НЕ видаляли — старі синхронізовані замовлення мають ці значення з legacy, лишаються для довідки, просто не редагуються через цю форму.

## Доступ для маркетингового агентства ("Експорт лідів")

Портовано з проміжного проєкту `erm-balkony--Only-for-marketing-page` (створений і одразу закритий; лишився лише як референс, read-only). Мета: зовнішнє маркетингове агентство отримує окремий акаунт, бачить ОДНУ сторінку — таблицю лідів з фільтром по датах — і може скачати її як CSV. Дашборд з графіками з того проєкту свідомо НЕ переносили (рішення користувача — лише таблиця + CSV).

Ключова відмінність від оригіналу: той проєкт читав СТАРУ базу напряму (плоска таблиця `leads` через окреме з'єднання); тут все читається з НАШОЇ бази (`leads` + `lead_marketing_data` + `clients` + `orders`) — сторінка переживе вивід старої CRM з експлуатації. Друга відмінність: в оригіналі CSV-маршрути були взагалі БЕЗ авторизації — тут обидва ендпоінти перевіряють той самий permission, що і сторінка.

- **`app/Services/Leads/LeadExportQueryService.php`** — read-only запит (leftJoin clients / lead_marketing_data / останній order на лід — корельований підзапит, щоб кілька замовлень не дублювали рядок ліда) + спільний `applyDateFilters()` і довідники лейблів ("цільовий"/статус-група виводяться з нових `stage`/`status`, семантика віддзеркалює `LeadsSyncMapper::STATUS_MAP`).
- **`app/Filament/Pages/Marketing/LeadExport.php`** — сторінка з таблицею (група "Маркетинг"), доступ `can('View:LeadExport')`. `shouldRegisterNavigation()` навмисно НЕ перевизначено — інакше сторінка зникла б з `NavigationCatalog::discover()` і її не можна було б керувати через "Бокову панель".
- **`app/Http/Controllers/Exports/LeadExportController.php`** + маршрути `/lead-export` (сторінка-лоадер, бо стрім 14k+ лідів триває кілька секунд) і `/lead-export/csv` (streamDownload, чанки по 500, UTF-8 BOM для Excel). Фільтр дат передається query-параметрами і застосовується тим САМИМ `applyDateFilters()` — файл завжди відповідає тому, що на екрані.
- **Міграція `create_marketing_agency_role`** — роль `marketing_agency` з рівно одним permission `View:LeadExport` (створюється тут же, `shield:generate` пізніше його просто перевикористає) + рядок у `navigation_settings`, бо не-адмінські ролі мають порожній сайдбар за замовчуванням і агентство інакше логінилось би в пустий інтерфейс.
- Побічне виправлення: `bootstrap/app.php` → `redirectGuestsTo('/admin/login')` — без цього будь-який `auth`-маршрут поза панеллю кидав гостям `RouteNotFoundException` (дефолтний редірект Laravel цілиться в неіснуючий route `login`).

Після міграції: створити користувача для агентства і призначити роль `marketing_agency`. Не забути `php artisan shield:generate --all --panel=admin`.

## "Документація сторінки" (PageDoc) — довідка прямо в адмінці

Окремий, наскрізний модуль (не специфічний для лідів) — точка істини для користувачів про те, як працювати з конкретною сторінкою/полем, що зберігається в БД і редагується прямо в адмінці, а не в зовнішніх документах.

- `page_docs` (page_key, section_key, title, content (HTML з RichEditor), sort_order, updated_by) — одна сторінка може мати кілька іменованих блоків (наприклад, для лідів зараз є лише блок `statuses`; пізніше можна додати ще, наприклад, по кожному складному полю окремо).
- Редагувати можуть тільки ролі `super_admin` і `founder` (нова роль, без жодних інших прав — створена саме для цього; засновник компанії не повинен мати повний технічний доступ, лише цю функцію). Перевірка — `Gate::define('manage-page-docs', ...)` в `AppServiceProvider`.
- Інші користувачі бачать тільки відрендерений HTML (через `Filament\Forms\Components\RichEditor\RichContentRenderer` — НІКОЛИ не виводити збережений HTML напряму через `{!! !!}` без цього рендерера, так каже сама документація Filament з безпеки).
- Редактор — `Filament\Forms\Components\RichEditor` (Tiptap), обмежений мінімальним набором кнопок (жирний/курсив/підкреслення, заголовки h2/h3, списки, цитата, посилання) — саме "блокнот з мінімальним форматуванням", як просив користувач, не повноцінний текстовий процесор.

**Як підключити до нової сторінки** (зразок — `EditLead`):
1. `use HasPageDocs;` (з `App\Filament\Concerns\HasPageDocs`) у класі сторінки.
2. Перевизначити `public function getFooter(): ?View { return $this->renderPageDoc('сторінка', 'секція', 'Заголовок за замовч.'); }` — Filament сам рендерить `getFooter()`, окремий `$view` не потрібен.
3. `renderPageDoc()` самостійно тягне `PageDoc`/перевіряє права і передає у `filament.components.page-doc` готові змінні (`$doc`, `$canManage`, `$action`) — партіал НІЧОГО не викликає через `$this`, бо `getFooter()` повертає окремий View-об'єкт, де `$this` вже не вказує надійно на компонент сторінки (на відміну від `@include` в тому самому файлі).

**Де саме живе документація лідів — важливе виправлення.** Спочатку блок "Статуси заявок" був над таблицею на сторінці списку лідів — користувач сам це виправив: список має бути просто таблицею, а документація (і взагалі будь-яка детальна інформація по конкретному ліду) належить сторінці ОКРЕМОГО ліда. Це призвело до ширшої архітектурної зміни:

**Лід тепер створюється в модалці, але переглядається/редагується на окремій сторінці.** `LeadResource` тепер реєструє і `index`, і `edit` (`EditLead extends EditRecord`) — реєстрація `edit`-сторінки автоматично перемикає поведінку `EditAction` з модалки на навігацію (той самий механізм, що й у `SupplierResource`, лише в інший бік). Причина: лід — це сутність, до якої повертаються (нагадування, історія дзвінків — плануються, ще не зроблено), тож вона заслуговує на повноцінну сторінку, а не модальне вікно.

Побічно виявлений і виправлений баг: правило перевірки дублю заявки за день (`Lead::where('client_id', ...)->whereDate('created_at', today())`) при редагуванні існуючого ліда натикалось саме на себе і блокувало збереження — виправлено, тепер виключає поточний запис (`whereKeyNot($record->getKey())`) коли `$record` не null (тобто це edit, не create).

## Рольова бокова панель + перемикання ролі для тестування

Ціль: дати можливість тестувати інтерфейс "очима" будь-якої ролі без створення окремих тестових акаунтів, і дати змогу налаштовувати, що саме бачить кожна роль у боковому меню — все редагується прямо в адмінці, а не хардкодом.

- **`users.active_role`** (nullable string) — яку зі своїх РЕАЛЬНИХ ролей користувач зараз "переглядає". Зберігається в БД (не в сесії) — свідомо, щоб перемикання переносилось між пристроями. `User::getActiveRoleName()` — повертає `active_role`, якщо він валідний (`hasRole()`), інакше першу реальну роль користувача.
- Перемикач — у верхньому меню користувача (де аватар): по одному пункту на КОЖНУ роль, що існує в системі, але `->visible()` показує лише ті, які реально є в поточного юзера (`AdminPanelProvider::roleSwitcherMenuItems()`). Клік оновлює `active_role` і перезавантажує сторінку.
- **`App\Services\Navigation\NavigationCatalog::discover()`** — інтроспектує `Filament::getPanel('admin')->getResources()`/`getPages()`, повертає природний (закодований) список пунктів меню з лейблом/групою/sort. Це і забезпечує "автоматичне підхоплення нових сторінок/ресурсів" — нічого вручну не реєструється.
- **`navigation_settings`** (role, item_key, group_label, group_sort, item_sort, is_active) — лише ПЕРЕВИЗНАЧЕННЯ для ролі; відсутній рядок = використовується дефолт з коду. `item_key` — повний FQCN класу Resource/Page (стабільний, доки клас не перейменовано/не переміщено).
- **`App\Services\Navigation\NavigationResolver`** — мердж каталогу з `navigation_settings` для конкретної ролі: `resolveForEditor()` (повний список, разом з неактивними — для сторінки налаштувань) і `resolveForSidebar()` (тільки активні, згруповані/відсортовані — для живого сайдбару).
- **`AdminPanelProvider::buildNavigationGroups()`** підключено через `->navigation(Closure)`. Важливий нюанс Filament: щойно передаєш Closure в `->navigation()`, автоматична побудова з `discoverResources`/`discoverPages`/`->navigationItems()` повністю вимикається — тому ручний шорткат "Додати заявку" (раніше — простий `->navigationItems([...])`) тепер збирається ВСЕРЕДИНІ цього ж Closure.

### Сторінки налаштувань ("Налаштування" → "Інтерфейс" → "Бокова панель")

`App\Filament\Pages\Settings\InterfaceSettings` — точка входу (поки один пункт-посилання, навмисно зроблено списком, бо очікується, що сюди додадуться інші налаштування інтерфейсу). `App\Filament\Pages\Settings\SidebarSettings` (`$shouldRegisterNavigation = false` — доступна лише за посиланням з Interface, не дублюється в меню) — вибір ролі зверху (`wire:model.live`) + таблиця `NavigationResolver::resolveForEditor($role)` з редагованими колонками (`group_sort`, `item_sort` — числові інпут-колонки; `is_active` — toggle), кожна з `->updateStateUsing()` напряму в `NavigationSetting::updateOrCreate()` (бо записи таблиці — не Eloquent-моделі, а синтетичний масив із каталогу — той самий трюк, що в `SyncOverview`). Сортування: активні зверху (в налаштованому порядку), неактивні — знизу.

**Оновлено після першого тестування користувачем:**
- Числові поля порядку → кнопки-стрілки (`Action` у `recordActions()`: "Група вище/нижче", "Пункт вище/нижче"). Клік одразу перенумеровує ВСІ елементи групи/списку послідовно (0,1,2,...) — не "своп" двох сирих значень, бо більшість пунктів мають однаковий sort=0 з коду і простий своп був би no-op.
- Активність (`is_active`) тепер НЕ зберігається миттєво — перемикач пише лише в локальний `$pendingActive` (Livewire-стан сторінки), і застосовується в БД тільки після кнопки "Застосувати зміни". Причина: миттєве збереження одразу пересортовувало таблицю (неактивні стрибали вниз), що плутало при клацанні кількох перемикачів підряд. Порядок (стрілки) і назва — навпаки, зберігаються миттєво, бо не викликають такого "стрибання".
- **`navigation_labels`** (item_key unique, label) — нова таблиця для перейменування пункту меню. На відміну від `navigation_settings`, це ГЛОБАЛЬНО (однакова назва для всіх ролей), не per-role.

**Друге оновлення — відмова від Filament Table на цій сторінці.** Плоска таблиця з групою, повтореною в кожному рядку, виявилась незрозумілою, і було неясно, чи спрацьовують кнопки переміщення групи. `SidebarSettings` тепер звичайна Livewire-сторінка (без `HasTable`/`InteractsWithTable`) із власним Blade: групи рендеряться як секції (заголовок групи — назва + стрілки переміщення групи; нижче — список пунктів, кожен зі своїми стрілками/назвою/перемикачем) — структура буквально повторює вигляд бокового меню. Перейменування групи тепер також можливе (`group_label`, per-role — те саме поле, що вже існувало в `navigation_settings`, просто раніше не мало редактора). Кожен `wire:click` викликає публічний метод сторінки, що завжди перечитує дані заново — тому будь-яка зміна видна одразу, без сумнівів "чи спрацювало".

**Третє оновлення — два окремі розділи замість сортування всередині групи.** Раніше неактивні пункти просто йшли в кінці СВОЄЇ групи (всередині того самого блоку) — недостатньо, бо верхня частина мала б на 100% повторювати бокову панель, а змішування з неактивними це порушувало. Тепер `SidebarSettings` рендерить дві окремі секції: **"Активна структура"** (`getActiveGroupedItems()` — лише активні, згруповані/відсортовані так само, як справжнє бокове меню — буквальна копія) і **"Вимкнені пункти"** (`getInactiveItems()` — плаский список усіх вимкнених, з назвою групи поруч як підказка, де воно з'явиться після увімкнення). Це й дає "подвійне сортування": активність — головний критерій розділення на секції, а група/позиція — лише ВСЕРЕДИНІ кожної секції.

**Окрема знахідка під час тестування — `npm run build` дійсно потрібен, моя рання відповідь була неправильною.** Власні Tailwind-класи в кастомних Blade-файлах (ця сторінка, документація сторінки тощо) НЕ входять у власний CSS-бандл Filament — вони компілюються через Vite/Tailwind цього проєкту. Правильний спосіб підключення — окрема тема `resources/css/filament/admin/theme.css`, яка імпортує `vendor/filament/filament/resources/css/theme.css` (звідси сам Filament бере `tailwindcss source(none)` + всі компоненти) і додає `@source` на `app/Filament` і `resources/views/filament`. Підключено через `->viteTheme(...)` в `AdminPanelProvider`. **Важливо: НЕ використовувати для цього `resources/css/app.css`** — там окремий власний `@import 'tailwindcss'`, і одночасне підключення обох призводить до двох конфліктних Tailwind-резетів і ламає всю адмінку (перевірено на практиці).

**Четверте оновлення — справжня причина "стрілки/перемикачі взагалі не працюють".** Це НЕ було стилями і НЕ було логікою сортування — `item_key` (повне ім'я класу, напр. `App\Filament\Resources\LeadResource`) ламався прямо у `wire:click="moveGroup('{{ $key }}', -1)"`: Livewire розбирає атрибут як JS-вираз, а JS у звичайному рядку **мовчки прибирає зворотний слеш** перед символом, який не є відомою escape-послідовністю (`\F`, `\R`, `\S` → просто `F`, `R`, `S`). В результаті на сервер прилітав ключ без жодного `\` (`AppFilamentResourcesLeadResource`), який ніде не знаходився — тому дія виконувалась, але результату не було. Знайдено напряму в БД (порівнянням збережених `item_key`). Виправлено через `Illuminate\Support\Js::from($key)` замість `'{{ $key }}'` в УСІХ `wire:click`/`wire:change` на цій сторінці — цей хелпер коректно екранує рядок для вставки в JS-контекст.

**П'яте оновлення — "Додати заявку" тепер теж керований пункт.** Цей ярлик ніколи не був справжнім Resource/Page, тому `NavigationCatalog::discover()` його не бачив, а `AdminPanelProvider` хардкодив окремим блоком прямо в `buildNavigationGroups()` — пункт існував у сайдбарі, але був невидимий і нерухомий у "Боковій панелі". Виправлено: `NavigationCatalog` тепер додає його як синтетичний запис (`key = 'quick-create-lead'`, без класу, без `\`), і він проходить через той самий конвеєр group/sort/active, що й усе інше. Хардкод у `AdminPanelProvider` прибрано повністю.

**Свідомо відкладено на наступну ітерацію**: візуальний прев'ю самого сайдбару прямо на сторінці налаштувань.

## Бокова панель ↔ права: «бачу = можу відкрити»

За явним запитом користувача видимість пункту меню зв'язана з правом перегляду (симетрично, включно з відкликанням):

- Кнопка "Застосувати зміни" на сторінці "Бокова панель" тепер, крім `navigation_settings`, викликає `SidebarPermissionSync::apply()` — увімкнення пункту видає ролі `ViewAny:<Model>`+`View:<Model>` (для Resource) або `View:<PageClass>` (для Page); вимкнення — відкликає.
- ТІЛЬКИ права перегляду. Дії (Create/Update/Delete/...) — як і раніше, вручну на сторінці "Ролі".
- `super_admin`/`founder` не синхронізуються (перший і так обходить перевірки, другий — навмисно мінімальна роль). Синтетичні пункти без класу (`quick-create-lead`) пропускаються.
- `php artisan sidebar:sync-permissions` — разове вирівнювання для ролей, налаштованих ДО появи цього зв'язку (саме через це "Менеджер" бачив меню, але ловив 403 на кліку). Безпечна для повторного запуску.

ВАЖЛИВО пам'ятати: перемикач ролей у верхньому меню міняє лише ВИГЛЯД сайдбару, а не права — super_admin під будь-якою "переглядуваною" роллю проходить усі перевірки. Тестувати реальні доступи можна лише справжнім акаунтом з цією роллю.

**Сторінки (Page) — заборонено за замовчуванням.** Filament захищає політиками лише Resources; окрема Page без власного `canAccess()` відкрита будь-якому залогіненому користувачу за прямим URL (так Менеджер відкрив "Аналітику рахунків", якої не мав у меню). Тому всі кастомні сторінки використовують трейт `App\Filament\Concerns\RequiresViewPermission` (`canAccess()` → `can('View:<ClassBasename>')`) — рівно той permission, який видає/відкликає SidebarPermissionSync. Винятки: `Dashboard` (домашній роут, мусить бути доступний усім — сам редіректить), `SyncTableDetail` (прихована детальна сторінка — успадковує право `View:SyncOverview`), `SidebarSettings` (не в каталозі меню → фактично лише super_admin). **Кожна НОВА сторінка повинна одразу отримувати цей трейт** — інакше вона знову буде відкрита всім за URL.

## Домашня сторінка = перший пункт бокового меню

Стандартну Filament-інфопанель замінено власним `App\Filament\Pages\Dashboard`: при відкритті він переадресовує користувача на ПЕРШИЙ пункт бокової панелі його активної ролі (`NavigationResolver::resolveForSidebar()`). Якщо дашборд свідомо стоїть першим у меню ролі — редіректу немає; якщо меню порожнє — дашборд лишається запасним варіантом. У `AdminPanelProvider` сток-Dashboard прибрано з `->pages()`. Міграція `rekey_dashboard_navigation_settings` перекейовує збережені налаштування панелі зі старого FQCN на новий.

## Принцип UX: «де змінюєш — там і зберігається»

**Жодних загальних кнопок "Зберегти" в інтерфейсі.**

Кожна зміна зберігається одразу в тому місці, де вона відбувається:

- **Select, Toggle, DatePicker** — `->live()` → зміна моментально потрапляє в Livewire і одразу зберігається в БД через `updated()` хук.
- **TextInput, Textarea** — `->lazy()` → зберігається при виході з поля (blur), не на кожен символ.
- **Складні дії** (upload файлу, прив'язка рахунку тощо) — через модальне вікно зі своєю кнопкою підтвердження всередині.

В `EditRecord`-сторінках обов'язково:
```php
protected function getFormActions(): array { return []; } // прибирає Save/Cancel знизу

public function updated(string $name): void {
    if (! str_starts_with($name, 'data.')) return;
    try {
        $this->save(shouldRedirect: false, shouldSendSavedNotification: false);
    } catch (\Illuminate\Validation\ValidationException) {}
      catch (\Throwable) {}
}
```

## Конвенції

- UI — українською. Код, коментарі в коді — англійською, з поясненням **чому**, не що (Filament-синтаксис самоочевидний).
- `ClientResource` свідомо без бізнес-логіки (без угод/замовлень) — "просте відображення" за прямим запитом, розширення — окрема майбутня задача.
