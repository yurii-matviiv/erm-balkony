# Документація: Аналітика рахунків

> Цей файл — жива документація сторінки `/admin/invoice-analytics`.  
> Оновлюй його при кожній зміні логіки. Тут зберігаються рішення, причини та визначення — те, що не видно з коду.

---

## 1. Призначення сторінки

Показує фінансовий стан компанії за обраний період:
- Скільки надійшло від клієнтів (готівка + безготівка)
- Скільки витрачено і на що
- Верифікація CRM-даних через реальні рухи по банківських рахунках ПриватБанк

Аудиторія: керівник компанії, бухгалтер. **Не** операційний інструмент менеджера.

---

## 2. Джерела даних

### A. Таблиця `order_payments` (CRM)
Оплати, синхронізовані зі старої CRM. Кожен запис прив'язаний до конкретного замовлення (`order_id`).

Ключові поля:
- `direction` — `income` (від клієнта) / `outgo` (нам треба платити)
- `payer_type` — `client`, `supplier`, `installer`, `gauger`, `office`, `expense`
- `category` — `salary`, `order`, `office`, `telephone`, `marketing`, `tax`, `recruitment`, `between_accounts`, NULL
- `payment_method` — `cash`, `cashless`, `card`, `courier`, `installer`
- `status` — тільки `received` рахується як реальна оплата
- **НЕ МАЄ `sub_category`** — ця колонка відсутня в таблиці. Всі синхронізовані записи мають NULL у sub_category

### B. Таблиця `expenses` (нова)
Загальні витрати без прив'язки до замовлення. Поки порожня (заповнюється вручну через майбутній Filament Resource).

Ключові поля: ті самі що `order_payments` + має `sub_category`.

### C. PrivatBank Business API (`acp.privatbank.ua`)
Реальні рухи по рахунках ФОП. Endpoint: `/statements/transactions`.

Ключові поля транзакції:
- `TRANTYPE` — `C` (credit, надходження) / `D` (debit, витрата)
- `SUM` — сума (рядок, напр. `"500.00"`)
- `OSND` — призначення платежу (текстовий коментар)
- `AUT_CNTR_NAM` — назва контрагента
- `AUT_CNTR_ACC` — IBAN контрагента
- `DAT_KL` — дата транзакції (`dd.MM.yyyy`)

⚠️ **Важливо:** `/balance` endpoint повертає `turnoverCredit`/`turnoverDebt` ТІЛЬКИ за СЬОГОДНІ, навіть якщо передати діапазон дат. Для агрегування за діапазон — завжди `/transactions` + ручне підсумовування.

## 3. Логіка KPI (три верхні картки)

```
Надходження = SUM(order_payments WHERE direction='income' AND status='received')
            + SUM(expenses WHERE direction='income' AND status='received')

Витрати     = SUM(order_payments WHERE direction='outgo' AND status='received'
                  AND category != 'between_accounts')   ← внутрішні перекази виключені
            + SUM(expenses WHERE direction='outgo' AND status='received')

Результат   = Надходження - Витрати
```

**Чому виключається `between_accounts`:**  
Записи `category='between_accounts'` — це зняття готівки з рахунку або перекази між рахунками компанії. Це не витрата бізнесу, а внутрішній рух. Якщо їх враховувати — витрати будуть завищені.

## 4. Таблиця надходжень

| Колонка | Запит |
|---|---|
| Готівка (CRM) | `order_payments` income WHERE payment_method='cash' |
| Безготівка (CRM) | `order_payments` income WHERE payment_method IN ('cashless','card') |
| Безготівка (ПриватБанк) | SUM транзакцій TRANTYPE='C' по всіх активних рахунках |
| Різниця | ПриватБанк - CRM безготівка |

**Що означає різниця:**
- **Позитивна** → банк показує більше надходжень ніж у CRM. Можливі причини: оплати ще не внесені в CRM, повернення, інші надходження не пов'язані з замовленнями.
- **Негативна** → CRM показує більше ніж банк. Малоймовірно при нормальній роботі.

Розбивка по рахунках ПриватБанк: показує і надходження (↑) і витрати (↓) — для швидкого орієнтування по кожному ФОП.

## 5. Таблиця витрат — групи та їх логіка

### 5.1 Зведена таблиця (вгорі)
Показує всі групи одразу. Назви груп з даними — клікабельні посилання-якорі (скрол до деталей нижче).

### 5.2 Групи витрат

#### Виробництво
```sql
order_payments WHERE direction='outgo' AND status='received'
  AND payer_type IN ('supplier', 'installer', 'gauger')
```
Підрядки: Постачальники / Монтажники / Замірники (за `payer_type`).  
Sub_category не доступна — синхронізована з замовлень, не з витрат.

#### Офіс, Телефонія, Маркетинг, Податки, Зарплата, Рекрутинг, Замовлення (інше)
Кожна група = **два джерела об'єднані**:
```sql
-- Джерело 1: expenses (нові записи, мають sub_category)
expenses WHERE direction='outgo' AND status='received' AND category=$group

-- Джерело 2: order_payments (старі записи прив'язані до замовлень)
order_payments WHERE direction='outgo' AND status='received'
  AND payer_type IN ('office','expense') AND category=$group
```
Агрегація по `sub_category` для відображення підрядків. Для Джерела 2 sub_category завжди NULL → підпадає під бакет `'other'` → відображається як `—`.

#### Некатегоризовано (якщо є)
Ловить все, що не потрапило в жодну групу. Якщо ця група показує суму — це сигнал перевірити дані.

### 5.3 Деталізовані таблиці (нижче)
Одна секція на кожну групу з даними. Показує кожен окремий платіж: дата, контрагент/опис, коментар, суми.

## 6. PrivatBank cross-checks

### Google Ads (група Маркетинг)
```
Фільтр транзакцій: TRANTYPE='D' AND OSND ILIKE '%google%'
```
Показує: CRM-сума за Google vs ПриватБанк-сума. Різниця — тривожний сигнал, якщо > 0.

### Податки ФОП (група Податки)
```
Фільтр транзакцій: TRANTYPE='D' AND (
    lower(OSND) CONTAINS 'єдиний податок' OR
    lower(OSND) CONTAINS 'єсв' OR
    lower(OSND) CONTAINS 'єдиний соціальний внесок' OR
    lower(OSND) CONTAINS 'пдфо' OR
    lower(AUT_CNTR_NAM) CONTAINS 'казначейство' OR
    lower(AUT_CNTR_NAM) CONTAINS 'дпс' OR
    lower(AUT_CNTR_NAM) CONTAINS 'державна податкова'
)
```
Показує: суму податкових платежів по кожному рахунку ПриватБанк.  
Це **не** дублює CRM-суму — це окремий погляд через банк. Логіка: в CRM вносяться податки вручну, в банку вони видні автоматично. Порівняння допомагає переконатись, що всі виплати внесені в CRM.

## 7. Відомі обмеження та технічний борг

### sub_category відсутня в order_payments
Причина: `OrderPaymentsSyncMapper` не синхронізував це поле (таблиця `order_payments` не має колонки `sub_category`).  
Наслідок: всі старі витрати типу office/salary/etc. в деталях показуються як `—` (не розподілені по підкатегоріях).  
Рішення: додати міграцію + колонку + ресинхронізацію. Поки не критично — коментар у платежі, як правило, пояснює суть.

### expenses таблиця порожня
Нова таблиця без даних — весь mass у `order_payments`. Майбутній Filament Resource для expenses заповнить її.

### Множинні API-виклики до ПриватБанк
Зараз: 3 окремих набори запитів (дохід, Google Ads, Податки) × N рахунків.  
Бажано: один запит на рахунок, кешований на рівні сервісу, з трьома фільтрами на ньому.  
Поки прийнятно (3 рахунки = 9 запитів, кожен ~200ms).

## 8. Бізнес-правила (погоджені)

| Правило | Рішення |
|---|---|
| `between_accounts` | **Виключати** з витрат і KPI. Це внутрішній рух, не витрата. |
| `expenses` тільки outgo? | Ні — можливий income (напр. повернення москітної сітки). Обидва direction враховуються в KPI. |
| Готівка vs безготівка | `payment_method='cash'` → готівка; все інше (`cashless`, `card`, ...) → безготівка. |
| Статус `pending` | **Не враховується** — тільки `status='received'`. |
| ПриватБанк cross-check | Cross-check — це **інформація**, не додаткова витрата. До загальної суми не додається. |

## 10. Майбутні модулі (що сюди додаватиметься)

- **Filament Resource для `expenses`** — форма введення офісних/телефонних/etc. витрат
- **Автоімпорт Google Ads з ПриватБанк** — записує знайдені транзакції в `expenses` автоматично
- **Автоімпорт податків ФОП з ПриватБанк** — аналогічно
- **Зарплата** — окремий модуль, зараз лише синхронізовані записи з order_payments
- **Консолідація API-запитів** — один getTransactions() на рахунок з кешуванням
